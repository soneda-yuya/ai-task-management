### Question C1

## Clarification 2: バックエンド構成(最重要)

**背景**: Q7 で「クラウド BaaS(Firebase / Supabase)」を選ばれましたが、Q10 で「Go バックエンド」、Q11 で「バックエンドは GCP」と回答されています。Firebase/Supabase は "バックエンドそのものを置き換える" BaaS なので、Go サーバーと共存させる構成を明確にする必要があります。

### Question C2
バックエンドアーキテクチャはどれに最も近いですか?

A) **BaaS のみ**: Supabase(Postgres + Auth + Realtime)を使い、Go バックエンドは作らない。重い処理のみ Supabase Edge Functions で対応。
B) **BaaS + 薄い Go バックエンド**: Supabase を DB / 認証基盤として使い、Go サーバー(Cloud Run)は **AI エージェント統合 / GitHub 連携** など BaaS で扱えない処理のみを担当する。
C) **GCP ネイティブ + Go バックエンド**: Firebase Auth + Cloud SQL(Postgres)or Firestore を使い、すべての業務ロジックを Go バックエンド(Cloud Run)で処理。BaaS の自動 CRUD API には頼らない。
D) **Firebase フルスタック**: Firebase(Auth + Firestore + Cloud Functions)でバックエンドを構築、Go は使わない。
