## 2. プロジェクトビジョン(長期)

複数の AI エージェントが**タスク管理(カンバンボード)**を起点に、以下を自律的に行う SaaS 型アプリケーション:

1. ユーザーが**アイデア / やりたいことを自由に入力** → AI が**対話形式で質問して深掘り** → AI が**階層的なタスク構造(親タスク + サブタスク)を提案** → **ユーザーが編集・承認してから正式タスク化**(もしくは手動でタスクを直接作成)
2. AI エージェントがタスクを分析 → 対応する **GitHub Issue を自動作成**
3. AI エージェントが該当リポジトリのコードを読解 → **実装して PR を作成**
4. **リスク評価エージェント**が PR のリスクを判定
5. リスクに応じて:
   - 低リスク → **自動マージ**
   - 中リスク → **CI 通過後に自動マージ**
   - 高リスク → **人間のレビュアーをアサイン**

## 4. 機能要件(Functional Requirements)

### 4.1 タスク管理機能(コア)

| ID | 要件 | 優先度 |
|---|---|---|
| FR-T-01 | ユーザーはタスクを作成、編集、削除できる(CRUD) | Must |
| FR-T-02 | ユーザーはタスクの完了/未完了を切り替えられる | Must |
| FR-T-03 | ユーザーはタスクに期限日を設定でき、期限に基づいたリマインダーを受け取る | Must |
| FR-T-04 | ユーザーはタスクに優先度(高/中/低)を設定できる | Must |
| FR-T-05 | ユーザーはタスクにカテゴリ/タグを付けられる | Must |
| FR-T-06 | ユーザーはサブタスク(ネスト構造)を作成できる | Must |
| FR-T-07 | ユーザーはタスクを他ユーザーにアサインできる(MVP では自分のみ) | Should |
| FR-T-08 | タスクはカンバンボード(To Do / Doing / Done)で表示される | Must |
| FR-T-09 | ユーザーはドラッグ&ドロップでタスクの状態を変更できる | Must |

### 4.3 AI エージェント機能

## 5. 非機能要件(Non-Functional Requirements)

### 5.1 パフォーマンス / スケール

| ID | 要件 |
|---|---|
| NFR-P-01 | 同時利用ユーザー数 **〜数千人**、タスク件数 **数十万件** を想定する |
| NFR-P-02 | タスク CRUD API の p95 レイテンシは **200ms 以下** |
| NFR-P-03 | カンバンボード表示の初期レンダリングは **1.5 秒以下**(Vercel Edge + 100 タスク時) |
| NFR-P-04 | AI エージェント実行は非同期、完了までは最大 **30 分**(長時間 PR 生成もあり得るため) |
| NFR-P-05 | バックエンドは水平スケール可能(Cloud Run の複数インスタンスで無停止スケール) |

### 5.2 可用性

| ID | 要件 |
|---|---|
| NFR-A-01 | 月間可用性 **99.5% 以上**(MVP)、将来的に 99.9% を目指す |
| NFR-A-02 | 計画メンテナンス時でも読み取り専用モードで動作可能な設計を考慮 |

### 5.3 セキュリティ(Security Baseline Extension: 強制)

| ID | 要件 |
|---|---|
| NFR-S-01 | 全 API は HTTPS のみ |
| NFR-S-02 | GitHub App のプライベートキーは Secret Manager で管理 |
| NFR-S-03 | AI エージェントが GitHub に書き込むアクションは監査ログに記録される |
| NFR-S-04 | ユーザー認証トークンはサーバーサイドで検証、XSS / CSRF 対策を実装 |
| NFR-S-05 | 依存パッケージの脆弱性は CI でスキャン(govulncheck、pip-audit など) |
| NFR-S-06 | AI エージェントの LLM への入力は PII(個人情報)を含まないようフィルタリング |
| NFR-S-07 | AI 生成コードは **サンドボックス環境** で実行・テストしてから PR 化する |

### 5.4 テスタビリティ(PBT Extension: 強制)

| ID | 要件 |
|---|---|
| NFR-T-01 | ビジネスロジック・データ変換・シリアライゼーションには**プロパティベーステスト**を適用 |
| NFR-T-02 | ユニットテストのカバレッジは **80% 以上** |
| NFR-T-03 | 統合テストは Cloud SQL のテストインスタンス(Testcontainers)で実行 |
| NFR-T-04 | AI エージェントのリスク判定ロジックは決定的テストケースでカバー |

### 5.5 保守性

| ID | 要件 |
|---|---|

## 6. 技術スタック(確定)

### フロントエンド

- **フレームワーク**: AI 推奨 → **Next.js 15+(App Router, TypeScript)**
  - 理由: React ベース(Q9 で React/Next.js が選択肢にあり)、Vercel とのネイティブ統合、SSR/SSG 対応で大規模スケール時の初期レンダリング最適化が容易
- **UI ライブラリ**: Tailwind CSS + shadcn/ui(または Radix UI)
- **ドラッグ&ドロップ**: dnd-kit(カンバンボード用)
- **状態管理**: TanStack Query(サーバー状態)+ Zustand(クライアント状態)
- **デプロイ**: Vercel

### バックエンド(中核 API)

- **言語**: Go 1.22+
- **Web フレームワーク**: `net/http` + `chi` ルーター、または Echo / Gin
- **ORM / DB アクセス**: `sqlc`(型安全、SQL ファースト)
- **認証**: Firebase Auth(Google / GitHub OAuth、JWT 検証を Go で実装)
- **非同期ジョブ**: Cloud Pub/Sub + Cloud Run Jobs
- **デプロイ**: Cloud Run(GCP)

### AI エージェントマイクロサービス

- **言語**: Python 3.12+
- **エージェント SDK**: **Claude Agent SDK**(公式)
- **LLM**: Claude Opus 4.6 / Sonnet 4.6 / Haiku 4.5 を用途別に使い分け
- **ツール**: GitHub API、ファイル操作、シェル実行(サンドボックス内)
- **デプロイ**: Cloud Run(GCP)

### データストア

- **プライマリ DB**: **Cloud SQL for PostgreSQL 16**
- **キャッシュ**: Memorystore for Redis(必要に応じて)
- **ファイルストレージ**: Cloud Storage(AI エージェントの中間生成物、ログ)

### インフラ / デプロイ

- **クラウド**: GCP(バックエンド、DB、AI サービス)+ Vercel(フロントエンド)
- **IaC**: Terraform
- **CI/CD**: GitHub Actions
- **コンテナ**: Docker + Cloud Run

### 外部連携

- **GitHub**: GitHub App(Webhook 経由)
- **認証**: Firebase Authentication(Google + GitHub OAuth、MVP 以降)

## 7. システム境界(System Boundary)

- **含まれる**: Web FE、Go BE API、AI Agent マイクロサービス、PostgreSQL、GitHub App、認証基盤
- **含まれない(外部依存)**: Anthropic API(Claude)、GitHub.com、Google OAuth、Firebase Auth の認証フロントエンド、CI(GitHub Actions 内のユーザー側リポジトリのテスト実行)

## 8. 制約条件 / 前提条件

- Claude API のレート制限・コストが AI エージェント実行数の上限となる
- GitHub App のインストール権限を各ユーザーが自分のリポジトリに付与する必要がある
- AI エージェントが PR を作成する対象リポジトリは、GitHub App がインストールされている必要がある
- ユーザーの Cloud SQL / Firebase / Vercel / Anthropic API / GitHub App のアカウントは事前準備必要

## 11. キー要件の要約(非技術者向け)

1. **ユーザーがアイデア/やりたいことを入力**すると、AI が対話で深掘りしてタスク候補を提案、ユーザー承認で正式タスク化(手動作成も可能)
2. **タスクはカンバンボード**で管理でき、期限・優先度・タグ・サブタスク・アサインなどの標準機能を備える
3. 各タスクに **GitHub リポジトリを紐付ける**と AI が Issue → 実装 → PR まで進める
4. **PR のリスクを AI が判定**し、安全なものは自動マージ、危険なものは人間にレビュー依頼
5. **大規模利用を前提とした設計**を最初から採用(Cloud Run + Cloud SQL)
6. **MVP は認証なし**、将来的に Google / GitHub OAuth で認証を追加できるよう設計を用意
