# ペルソナ (Personas)

**プロジェクト**: AI Task Manager
**作成日**: 2026-04-17
**情報レベル**: 標準(名前、役割、主ゴール、モチベーション、フラストレーション、技術リテラシー)
**選定**: P1 により A, B, C, D(外部エンドユーザー E は MVP で匿名動作するため対象外)

---

## Persona 1: 田中 健 — 個人開発者(Solo Developer)

| 項目 | 内容 |
|---|---|
| 役割 | フリーランスエンジニア |
| 主なゴール | 自分の副業プロジェクトで「タスク作成 → AI 実装 → PR マージ」の高速サイクルを回す |
| モチベーション | 手戻りを減らし、本業の合間に効率よく副業を進めたい。AI に実装を任せることで設計・仕様の思考時間を確保したい |
| フラストレーション | 仕様書作成と実装の分離がつらい、ひとりなのでレビューできる人がいない、手動で Issue を書くのが億劫 |
| 技術リテラシー | 高 — Go / TypeScript に慣れ、CLI・Git・GitHub Actions を日常利用 |
| 利用頻度 | 毎日 2〜3 時間 |
| 関連ストーリー | US-1.*, US-2.*, US-3.*(特に US-3.4 低リスク自動マージを多用) |

---

## Persona 2: 鈴木 結衣 — 小規模チームリーダー(Team Lead)

| 項目 | 内容 |
|---|---|
| 役割 | スタートアップのエンジニアリングチームリーダー(部下 4 名) |
| 主なゴール | チーム全体の生産性向上、AI 支援を定常運用に組み込み、メンバーのボトルネックを可視化する |
| モチベーション | 限られた人員で最大のアウトプットを出したい、AI との協業ワークフローをチームに浸透させたい |
| フラストレーション | メンバーの進捗把握に時間がかかる、レビューのキューが詰まる、複数リポジトリの状態が散らばる |
| 技術リテラシー | 高 — Web スタック全般に精通、アーキテクチャレベルの判断も行う |
| 利用頻度 | 毎日、MTG 前後やタスク割当時 |
| 関連ストーリー | US-1.*(アサイン含む), US-2.5(複数リポ), US-3.7(マージポリシー上書き), US-5.*(監視) |

---

## Persona 3: 佐藤 亮 — コードレビュアー(Code Reviewer)

| 項目 | 内容 |
|---|---|
| 役割 | シニアエンジニア / テックリード |
| 主なゴール | 高リスクな変更を確実にレビューし、コードベースの健全性を保つ |
| モチベーション | 低リスクな PR はロボットに任せて、自分の時間は価値の高いレビューと設計に集中したい |
| フラストレーション | 雑な PR や巨大 PR に時間を取られる、レビュー依頼の粒度や重要度がバラバラ |
| 技術リテラシー | 非常に高 — パフォーマンス・セキュリティ・DB スキーマ設計まで精通 |
| 利用頻度 | 通知ベースで不定期 |
| 関連ストーリー | US-3.3(リスク評価依存)、US-3.6(高リスク PR レビュー)、US-5.*(監査) |

---

## Persona 4: 山田 麻衣 — Organization 管理者(Org Admin)

| 項目 | 内容 |
|---|---|
| 役割 | DevOps エンジニア / エンジニアリングマネージャ |
| 主なゴール | GitHub Organization 配下のアプリ権限・セキュリティ・監査証跡を一元管理する |
| モチベーション | 自動化ツール起因のインシデントを防ぎ、監査要求に応えられる状態を維持したい |
| フラストレーション | 各 App の権限スコープ管理が煩雑、監査ログがシステム横断で散らばる、脆弱性追跡が後手に |
| 技術リテラシー | 非常に高 — クラウド IAM、SBOM、SIEM ツールなどに精通 |
| 利用頻度 | 週次でレビュー、インシデント発生時は集中利用 |
| 関連ストーリー | US-2.1(App インストール)、US-5.*(監査・監視)、US-6.*(セキュリティ全般) |
