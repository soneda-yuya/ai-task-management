# ユーザーストーリー (User Stories)

**プロジェクト**: AI Task Manager
**作成日**: 2026-04-17
**書式**: Connextra 形式(As a ... I want ... so that ...)+ Gherkin 受入条件(Given/When/Then)
**構造**: 機能ドメイン別(エピック)+ 各ドメイン内でペルソナを明示
**優先度**: MoSCoW(Must / Should / Could / Won't have for MVP)
**粒度**: 標準(1 ストーリー = 2〜5 日で実装可能)
**INVEST 準拠**: Independent / Negotiable / Valuable / Estimable / Small / Testable すべてを満たすよう設計

## Epic 1: タスク管理 (Task Management)

**目的**: カンバンスタイルのタスク管理機能を提供する
**主要ペルソナ**: 田中(Solo), 鈴木(Lead)

---

### US-1.1: タスクを作成する
**優先度**: Must | **ペルソナ**: Solo, Lead

**ストーリー**: **個人開発者 / チームリーダー**として、タイトル・説明・メタデータを含む**新しいタスクを作成**したい。それは**自分の作業または部下のアサイン対象を記録する**ためだ。

**受入条件**:
- **Given** ユーザーがカンバンボードを開いている、**When** 「タスク追加」ボタンを押しタイトルを入力する、**Then** タスクが「To Do」列の先頭に表示される
- **Given** タスクタイトルが空欄、**When** 保存を試みる、**Then** バリデーションエラーが表示され作成されない
- **Given** タイトル長が 200 文字を超える、**When** 保存を試みる、**Then** バリデーションエラーが表示される
- **Given** タスク作成リクエスト、**When** API が処理する、**Then** p95 レイテンシが 200ms 以下である (NFR-P-02)

---

### US-1.2: タスクを編集・削除する
**優先度**: Must | **ペルソナ**: Solo, Lead

**ストーリー**: **個人開発者 / チームリーダー**として、**既存タスクの内容を編集または削除**したい。それは**誤入力の修正や不要タスクの整理をする**ためだ。

**受入条件**:
- **Given** 既存のタスクを開く、**When** 説明を更新し保存、**Then** 変更が永続化され履歴に残る
- **Given** タスク削除を確定する、**When** 削除を実行、**Then** タスクがソフトデリートされ 30 日後に物理削除される
- **Given** 誤って削除した、**When** 30 日以内にゴミ箱から復元する、**Then** 元の列・位置に戻る

---

### US-1.3: タスクの完了/未完了を切り替える
**優先度**: Must | **ペルソナ**: Solo, Lead

**ストーリー**: **ユーザー**として、**タスクの完了状態をワンクリックで切り替え**たい。それは**進捗を即座に反映する**ためだ。

**受入条件**:
- **Given** 未完了タスク、**When** チェックボックスをクリック、**Then** 完了状態になり Done 列に移動する
- **Given** 完了タスク、**When** 再クリック、**Then** 未完了に戻り前回の列に戻る
- **Given** タスクに未完了のサブタスクが残っている、**When** 親を完了にする、**Then** 警告ダイアログで確認する

---

### US-1.4: タスクに期限とリマインダーを設定する
**優先度**: Must | **ペルソナ**: Solo, Lead

**ストーリー**: **ユーザー**として、**タスクに期限日を設定しリマインダーを受け取り**たい。それは**締切を守る**ためだ。

**受入条件**:
- **Given** タスクを開く、**When** 期限日を選択、**Then** タスクカードに期限が表示される
- **Given** 期限の 24 時間前、**When** システムが通知をトリガー、**Then** ユーザーに通知が届く
- **Given** 期限を過ぎた未完了タスク、**When** カンバンボードを開く、**Then** カードが赤色でハイライトされる

---

### US-1.5: タスクに優先度を設定する
**優先度**: Must | **ペルソナ**: Solo, Lead

**ストーリー**: **ユーザー**として、**タスクに優先度(高/中/低)を設定**したい。それは**重要度の高いタスクから着手する**ためだ。

**受入条件**:
- **Given** タスク編集画面、**When** 優先度ラジオを選択、**Then** カード左端の色インジケータが変わる(高=赤、中=黄、低=緑)
- **Given** 優先度でフィルタ、**When** フィルタを適用、**Then** 該当優先度のタスクのみ表示される

---

### US-1.6: タスクにタグを付ける
**優先度**: Must | **ペルソナ**: Solo, Lead

**ストーリー**: **ユーザー**として、**タスクに複数のタグを付与**したい。それは**カテゴリで分類・検索する**ためだ。

**受入条件**:
- **Given** タスク編集画面、**When** タグ入力フィールドに #bug と入力して Enter、**Then** タグが付与される
- **Given** 存在しないタグを入力、**When** 保存、**Then** 新しいタグが自動作成され候補に追加される
- **Given** タグでフィルタ、**When** サイドバーから #bug を選択、**Then** タグ付きタスクのみ表示される

---

### US-1.7: サブタスクを作成する
**優先度**: Should | **ペルソナ**: Solo, Lead

**ストーリー**: **ユーザー**として、**タスクの下に子タスク(サブタスク)を作成**したい。それは**大きな作業を小さなステップに分解する**ためだ。

**受入条件**:
- **Given** 親タスクを開く、**When** 「サブタスク追加」を押しタイトル入力、**Then** 子タスクがツリー表示される
- **Given** ネストは 2 階層まで、**When** 3 階層目を作成しようとする、**Then** 「サブタスクは 2 階層まで」エラーが出る
- **Given** 親タスクが完了した、**When** 完了後に子タスクが残る、**Then** US-1.3 の警告が表示される

---

### US-1.8: タスクをメンバーにアサインする
**優先度**: Should | **ペルソナ**: Lead

**ストーリー**: **チームリーダー**として、**タスクを特定のメンバーに割り当て**たい。それは**責任範囲を明確にする**ためだ。

**受入条件**:
- **Given** タスク編集画面、**When** アサイン対象をドロップダウンから選ぶ、**Then** タスクカードにアサイニーのアバターが表示される
- **Given** アサイニーでフィルタ、**When** 自分のアイコンを選択、**Then** 自分にアサインされたタスクのみ表示される
- **Given** MVP は単一ユーザー動作、**When** アサイン先選択 UI を開く、**Then** 自分のみが候補として表示される(Phase2 で拡張)

---

### US-1.9: カンバンボードでタスクを可視化する
**優先度**: Must | **ペルソナ**: Solo, Lead

**ストーリー**: **ユーザー**として、**タスクを To Do / Doing / Done の列で一覧**したい。それは**全体の進捗状況を一目で把握する**ためだ。

**受入条件**:
- **Given** カンバンボードを開く、**When** 初期表示される、**Then** 1.5 秒以内(100 タスク時)にレンダリングされる (NFR-P-03)
- **Given** 各列、**When** タスク数が表示される、**Then** 列のヘッダーに件数バッジが出る
- **Given** タスク 1000 件超、**When** ボードを開く、**Then** 仮想スクロールで滑らかに表示される

---

### US-1.10: ドラッグ&ドロップでタスクの状態を変える
**優先度**: Must | **ペルソナ**: Solo, Lead

**ストーリー**: **ユーザー**として、**タスクカードをドラッグ&ドロップで別の列に移動**したい。それは**状態変更を直感的に行う**ためだ。

**受入条件**:
- **Given** タスクを掴む、**When** 別の列にドロップする、**Then** 状態が更新され楽観的に UI が即反映される
- **Given** API が失敗した、**When** サーバーへの反映が失敗する、**Then** 元の位置に戻し、エラートーストを表示する
- **Given** 同時に複数ユーザーが同一タスクを動かした、**When** 競合が発生、**Then** 最後の更新が勝つ(Last-Write-Wins)

---

## Epic 2: GitHub リポジトリ連携 (GitHub Integration)

**目的**: GitHub App 経由でマルチリポジトリと連携し、タスクと Issue を同期する
**主要ペルソナ**: 田中, 鈴木, 山田(App インストール時)

---

### US-2.1: GitHub App をリポジトリにインストールする
**優先度**: Must | **ペルソナ**: Org Admin

**ストーリー**: **Organization 管理者**として、**本アプリの GitHub App を任意のリポジトリにインストール**したい。それは**AI エージェントに必要な権限を安全に付与する**ためだ。

**受入条件**:
- **Given** 本アプリの設定画面、**When** 「GitHub App をインストール」ボタンを押す、**Then** GitHub のインストール UI にリダイレクトされる
- **Given** GitHub でインストール承認、**When** リポジトリを選択して承認、**Then** アプリ側にインストール完了イベントが Webhook で届く
- **Given** インストール後、**When** アプリの連携リポ一覧を開く、**Then** 選択されたリポジトリが表示される
- **Given** Org Admin がインストールをアンインストールした、**When** Webhook を受信、**Then** 関連タスクは残るが AI 実行はブロックされる

---

### US-2.2: タスクにリポジトリを紐付ける
**優先度**: Must | **ペルソナ**: Solo, Lead

**ストーリー**: **ユーザー**として、**各タスクに GitHub リポジトリを紐付け**たい。それは**AI エージェントが対象リポジトリを特定できる**ためだ。

**受入条件**:
- **Given** タスク編集画面、**When** リポジトリドロップダウンから選ぶ、**Then** 連携済みリポのみが候補に出る
- **Given** 未連携のリポ、**When** 選ぼうとする、**Then** 「GitHub App を先にインストール」案内が出る
- **Given** タスクにリポが紐付く、**When** 保存、**Then** カードにリポ名バッジが表示される

---

### US-2.3: タスクから GitHub Issue を自動作成する
**優先度**: Must | **ペルソナ**: Solo, Lead

**ストーリー**: **ユーザー**として、**タスクから対応する GitHub Issue を自動生成**したい。それは**手動で Issue を書かずに済ませる**ためだ。

**受入条件**:
- **Given** リポ紐付け済みタスク、**When** 「Issue を作成」を押す、**Then** AI が Issue 本文を生成して GitHub に作成、Issue URL がタスクに記録される
- **Given** 既に Issue が作成済み、**When** 再度作成を試みる、**Then** 既存 Issue へのリンクを表示し重複作成を防ぐ
- **Given** GitHub API が失敗、**When** 作成が失敗、**Then** エラー内容を表示しリトライボタンを提供する

---

### US-2.4: Issue 状態をタスクと双方向同期する
**優先度**: Should | **ペルソナ**: Solo, Lead

**ストーリー**: **ユーザー**として、**GitHub 上の Issue の状態変更をタスクへ同期**したい。それは**GitHub とアプリの状態をずれなく保つ**ためだ。

**受入条件**:
- **Given** 外部で Issue がクローズされた、**When** Webhook を受信、**Then** 対応タスクが Done 列に移動する
- **Given** アプリ側でタスクを Done にした、**When** 状態更新、**Then** GitHub Issue が closed になる
- **Given** 双方向の同時更新、**When** 衝突検出、**Then** タイムスタンプ新しい方を採用しログに記録する

---

### US-2.5: 複数リポジトリを一元管理する
**優先度**: Must | **ペルソナ**: Lead

**ストーリー**: **チームリーダー**として、**複数のリポジトリに跨るタスクを一画面で管理**したい。それは**プロジェクト横断の進捗を把握する**ためだ。

**受入条件**:
- **Given** 3 つ以上のリポが連携済み、**When** ダッシュボードを開く、**Then** リポ別のタスク件数サマリが表示される
- **Given** リポフィルタ、**When** 特定リポを選択、**Then** そのリポのタスクのみ表示
- **Given** 同一タスク ID が別リポで重複しない、**When** 作成時、**Then** システム全体でユニーク ID を保証する

---

## Epic 3: AI エージェント自律実行 (AI Agent Autonomous Execution)

**目的**: AI エージェントがタスクを受け取り、Issue 生成 → 実装 → PR 作成 → リスク判定 → 自動マージ or レビュアー割当までを自律実行する
**主要ペルソナ**: 田中, 鈴木, 佐藤

---

### US-3.1: AI が Issue 本文を自動生成する
**優先度**: Must | **ペルソナ**: Solo, Lead

**ストーリー**: **ユーザー**として、**タスクの情報から AI が詳細な Issue 本文を生成**したい。それは**仕様を自分で書く手間を省く**ためだ。

**受入条件**:
- **Given** タスク情報(タイトル、説明、タグ、関連リポ)、**When** AI が生成、**Then** 背景・要求事項・受入条件を含む Issue 本文が得られる
- **Given** 生成された Issue 本文、**When** ユーザーが確認画面を見る、**Then** 送信前に編集できる
- **Given** PII を含む入力、**When** AI が検知、**Then** マスク処理後に送信される (US-3.10 参照)

---

### US-3.2: AI がコード実装の PR を作成する
**優先度**: Must | **ペルソナ**: Solo, Lead

**ストーリー**: **ユーザー**として、**AI が Issue の要求に沿って実装を行い、PR として提出**することを望む。それは**実装作業を自動化する**ためだ。

**受入条件**:
- **Given** Issue と対象リポ、**When** AI エージェントが実装開始、**Then** 新しい feature ブランチで変更を加え PR を作成する
- **Given** AI が実装中、**When** リポをサンドボックス環境にクローンする、**Then** ホストから隔離された環境で実行される (NFR-S-07)
- **Given** 失敗(ビルドエラー等)、**When** AI が失敗を検知、**Then** エラー内容と次の試行方針をタスクに記録してユーザーに通知する
- **Given** PR 作成完了、**When** タスク UI を確認、**Then** PR URL、差分行数、変更ファイルリストが表示される

---

### US-3.3: リスク評価エージェントが PR リスクを判定する [System Story]
**優先度**: Must | **ペルソナ**: (システム内部)

**ストーリー**: **システム**として、**作成された PR の変更量・影響範囲・セキュリティ影響からリスクレベルを判定**したい。それは**自動マージ可否を適切に決める**ためだ。

**受入条件**:
- **Given** 新規作成された PR、**When** リスク評価サブエージェントが評価、**Then** 低 / 中 / 高のいずれかを返す
- **Given** 差分行数が 50 行以下 AND セキュリティ関連ファイル(auth, secrets)を含まない、**When** 判定、**Then** 低リスクに分類される
- **Given** 依存ライブラリの追加 OR DB マイグレーション、**When** 判定、**Then** 中リスク以上に分類される
- **Given** 認証・権限・API 公開範囲の変更、**When** 判定、**Then** 高リスクに分類される
- **Given** 判定結果、**When** PR にコメント、**Then** リスクレベルと根拠の要約が記載される

---

### US-3.4: 低リスク PR が自動マージされる
**優先度**: Must | **ペルソナ**: Solo, Lead

**ストーリー**: **ユーザー**として、**低リスクと判定された PR が自動マージされる**ことを望む。それは**安全な変更を手動作業なしで取り込める**ためだ。

**受入条件**:
- **Given** リスク=低の PR、**When** マージポリシーが「自動」または未指定、**Then** 即座にマージされる
- **Given** マージ完了、**When** タスクを確認、**Then** Done 列に移動しマージコミット SHA が記録される
- **Given** ブランチ保護ルールに違反、**When** マージを試みる、**Then** 自動マージを中止しレビュアーアサインにフォールバックする

---

### US-3.5: 中リスク PR が CI 通過後に自動マージされる
**優先度**: Must | **ペルソナ**: Solo, Lead

**ストーリー**: **ユーザー**として、**中リスクな PR は CI(テスト・lint)が通過してから自動マージ**されることを望む。それは**品質を担保しつつ自動化の恩恵を受ける**ためだ。

**受入条件**:
- **Given** リスク=中の PR、**When** CI 実行中、**Then** マージ待機状態として UI 表示される
- **Given** CI 全ジョブが成功、**When** ジョブ結果 Webhook 受信、**Then** 自動マージされる
- **Given** CI が失敗、**When** Webhook 受信、**Then** マージを中止し、失敗ログと共にタスクへ通知する

---

### US-3.6: 高リスク PR に人間のレビュアーがアサインされる
**優先度**: Must | **ペルソナ**: Reviewer

**ストーリー**: **コードレビュアー**として、**高リスクと判定された PR にはレビュー依頼が自動で届く**ことを望む。それは**重要な変更を見逃さない**ためだ。

**受入条件**:
- **Given** リスク=高の PR、**When** 判定完了、**Then** 事前設定されたレビュアー(または CODEOWNERS)がアサインされる
- **Given** レビュアーに通知、**When** GitHub 通知と本アプリ UI の両方、**Then** リスク根拠のサマリ付きで届く
- **Given** 高リスク PR、**When** 自動マージは実行されない、**Then** レビュアーの承認まで待機状態を保つ

---

### US-3.7: タスクごとにマージポリシーを上書きする
**優先度**: Should | **ペルソナ**: Lead

**ストーリー**: **チームリーダー**として、**タスクごとに「常に手動マージ」「常に自動」「CI 依存」を指定**したい。それは**重要なタスクはリスクに関係なく人間が確認したい**などのポリシーを反映するためだ。

**受入条件**:
- **Given** タスク編集画面、**When** マージポリシーを「常に手動」に設定、**Then** リスクに関係なくレビュアーにアサインされる
- **Given** ポリシー未指定、**When** リスクレベルに基づき判定、**Then** US-3.4/3.5/3.6 のフローに従う
- **Given** チーム全体のデフォルトポリシー、**When** 設定画面で変更、**Then** 新規タスクに適用される(既存は変えない)

---

### US-3.8: AI 実行の進捗・ログを GUI で追跡する
**優先度**: Must | **ペルソナ**: Solo, Lead

**ストーリー**: **ユーザー**として、**AI エージェントの実行ステップを GUI で確認**したい。それは**失敗時の原因特定や進捗把握ができる**ためだ。

**受入条件**:
- **Given** AI 実行中のタスク、**When** 詳細画面を開く、**Then** 「Issue 生成 → 分析 → 実装 → テスト → PR → リスク判定」のステップが時系列表示される
- **Given** 各ステップ、**When** 展開する、**Then** LLM 呼び出しログ、ツール実行結果、経過時間が確認できる
- **Given** 実行失敗、**When** ステップを展開、**Then** エラースタックと推奨対応が表示される

---

### US-3.9: AI 実行は非同期で UI をブロックしない [System Story]
**優先度**: Must | **ペルソナ**: (システム内部)

**ストーリー**: **システム**として、**AI エージェント実行をバックグラウンドジョブ化**したい。それは**長時間処理中も UI が操作可能である**ためだ。

**受入条件**:
- **Given** AI 実行リクエスト、**When** API が受け付ける、**Then** Pub/Sub にジョブを投入し即座に 202 Accepted を返す
- **Given** 最大実行時間、**When** 30 分を超える、**Then** タイムアウトとしてエラー記録し通知する (NFR-P-04)
- **Given** 同一タスクに対する並行実行、**When** 新規実行リクエスト、**Then** ロックにより二重起動を防ぐ

---

### US-3.10: AI が PII を検知してフィルタする [System Story, Security]
**優先度**: Must | **ペルソナ**: (システム内部)

**ストーリー**: **システム**として、**AI への入力から PII(メール・電話番号・クレカ等)を検知・マスク**したい。それは**ユーザーのプライバシーを保護する**ためだ。

**受入条件**:
- **Given** タスク本文に PII 候補文字列、**When** AI 送信前、**Then** 正規表現 + LLM 軽量分類器で検知しマスクする
- **Given** マスク処理された入力、**When** ユーザーがプレビュー、**Then** 置換箇所が UI に表示される
- **Given** PII 検知、**When** 検知イベント、**Then** 監査ログに記録する(マスク後の文字列のみ)

## Epic 4: 認証 / アカウント (Authentication & Account)

**目的**: MVP は匿名シングルユーザー動作、将来の OAuth 追加に備えた設計
**主要ペルソナ**: 全ペルソナ(Phase2)

---

### US-4.1: MVP では匿名シングルユーザーで動作する [System Story]
**優先度**: Must (MVP) | **ペルソナ**: (システム内部)

**ストーリー**: **システム**として、**MVP フェーズでは認証なしの匿名シングルユーザーモードで動作**したい。それは**初期リリースの摩擦を下げつつ後付け可能にする**ためだ。

**受入条件**:
- **Given** MVP モード、**When** アプリにアクセス、**Then** 自動で固定の "default-user" としてログインされる
- **Given** 認証抽象層、**When** 実装、**Then** IAuthProvider インターフェース経由で AnonymousProvider が動作する
- **Given** Cloud Run の Ingress、**When** 外部アクセス、**Then** 特定 IAM 設定または Cloud IAP で保護される

---

### US-4.2: Google OAuth でログインする
**優先度**: Should (Phase2) | **ペルソナ**: Solo, Lead, Reviewer

**ストーリー**: **ユーザー**として、**Google アカウントでサインイン**したい。それは**別途パスワードを管理せず迅速にログインする**ためだ。

**受入条件**:
- **Given** Firebase Auth 経由、**When** Google ボタンを押す、**Then** Google の認証画面が開く
- **Given** 認証完了、**When** リダイレクト、**Then** JWT を取得しアプリに保存、以降の API 呼び出しに Authorization ヘッダーで付与される
- **Given** JWT 期限切れ、**When** リフレッシュトークンでの更新、**Then** 自動更新される

---

### US-4.3: GitHub OAuth でログインする
**優先度**: Should (Phase2) | **ペルソナ**: Solo, Lead, Reviewer

**ストーリー**: **ユーザー**として、**GitHub アカウントでサインイン**したい。それは**GitHub App 操作と認証を同じアカウントに紐付ける**ためだ。

**受入条件**:
- **Given** Firebase Auth 経由、**When** GitHub ボタンを押す、**Then** GitHub OAuth 認可画面が開く
- **Given** 認証成功、**When** アプリに戻る、**Then** GitHub ユーザー名とアバターが表示される
- **Given** 複数プロバイダ、**When** 同じメールで Google と GitHub の両方を試す、**Then** Firebase Auth のアカウントリンク機能で統合される

---

### US-4.4: 認証基盤を最初から組み込む [System Story]
**優先度**: Must (MVP で設計確保) | **ペルソナ**: (システム内部)

**ストーリー**: **システム**として、**認証基盤(Firebase Auth)を MVP 時点で組み込んでおく**ことを望む。それは**Phase2 で認証方式を切り替える際に大規模な書き換えを避ける**ためだ。

**受入条件**:
- **Given** 全 API エンドポイント、**When** リクエスト到達、**Then** AuthMiddleware が Context にユーザー ID を注入する
- **Given** MVP の AnonymousProvider、**When** デフォルトユーザーで動作、**Then** ユーザー ID = "default-user" が注入される
- **Given** Phase2 で切り替え、**When** 実装 = FirebaseAuthProvider、**Then** 既存の API コードはほぼ無変更で動作する

---

## Epic 5: 監視・監査 (Observability & Audit)

**目的**: 運用状態・AI 動作・セキュリティイベントを可視化・記録する
**主要ペルソナ**: 鈴木(運用), 山田(監査)

---

### US-5.1: AI 実行結果の監査ログ
**優先度**: Must | **ペルソナ**: Org Admin

**ストーリー**: **Organization 管理者**として、**AI エージェントが実行した全操作(Issue 作成 / PR 作成 / マージ)を監査ログとして参照**したい。それは**不適切な自動変更を追跡・特定する**ためだ。

**受入条件**:
- **Given** AI が GitHub に書き込みアクション、**When** 実行、**Then** 監査ログにタイムスタンプ・対象リポ・PR 番号・リスクレベルが記録される (NFR-S-03)
- **Given** 監査ログ検索画面、**When** ユーザー・期間・アクション種別でフィルタ、**Then** 結果が表示される
- **Given** 監査ログ、**When** エクスポート、**Then** CSV / JSON でダウンロード可能

---

### US-5.2: GitHub App アクションの監査ログ
**優先度**: Must | **ペルソナ**: Org Admin

**ストーリー**: **Organization 管理者**として、**GitHub App のインストール・権限変更・アンインストールを追跡**したい。それは**セキュリティ監査要求に応える**ためだ。

**受入条件**:
- **Given** App の Webhook イベント(installation, installation_repositories)、**When** 受信、**Then** 監査ログに記録される
- **Given** 権限変更イベント、**When** 検出、**Then** Org Admin に通知が送られる

---

### US-5.3: エラー・レイテンシのメトリクス可視化
**優先度**: Should | **ペルソナ**: Lead, Org Admin

**ストーリー**: **チームリーダー / Organization 管理者**として、**システムのエラー率・API レイテンシ・AI 成功率を可視化するダッシュボード**を見たい。それは**運用状態を把握し、劣化を早期検知する**ためだ。

**受入条件**:
- **Given** Cloud Monitoring ダッシュボード、**When** 開く、**Then** 直近 24h のエラー率・p95 レイテンシ・AI 実行成功率が表示される (NFR-M-03)
- **Given** アラート閾値超過(例: エラー率 > 5%)、**When** 発生、**Then** メール / Slack に通知される

## Epic 6: 非機能要件 (Security & PBT — Independent Stories)

**目的**: Security / PBT 拡張機能の要件を独立ストーリー化し、監査性・トレーサビリティを確保する
**主要ペルソナ**: 山田(監査), 佐藤(レビュー対象)

---

### US-6.1: 全 API は HTTPS のみで提供される [System, Security]
**優先度**: Must | **ペルソナ**: (システム内部)

**ストーリー**: **システム**として、**全 API を HTTPS でのみ公開**したい。それは**通信盗聴を防ぐ**ためだ。

**受入条件**:
- **Given** Cloud Run のデプロイ設定、**When** HTTP アクセスを試みる、**Then** HTTPS に強制リダイレクトされる
- **Given** TLS 証明書、**When** Google-managed certificate を利用、**Then** 自動更新される

---

### US-6.2: GitHub App プライベートキーは Secret Manager で管理される [System, Security]
**優先度**: Must | **ペルソナ**: (システム内部)

**ストーリー**: **システム**として、**GitHub App の秘密鍵を環境変数やファイルで持たず Secret Manager から取得**したい。それは**鍵漏洩リスクを低減する**ためだ。

**受入条件**:
- **Given** Cloud Run のサービスアカウント、**When** Secret Manager の secret を読み取る権限のみ付与、**Then** 必要時にシークレットをメモリに展開する
- **Given** キーのローテーション、**When** 新しい鍵を Secret Manager に追加、**Then** アプリは自動で新鍵を使用(バージョン `latest` 参照)

---

### US-6.3: AI 生成コードはサンドボックスで実行・テストされる [System, Security]
**優先度**: Must | **ペルソナ**: (システム内部)

**ストーリー**: **システム**として、**AI が生成したコードをホストから隔離されたサンドボックス(短命コンテナ)で実行**したい。それは**悪意または事故で本体環境が汚染されるのを防ぐ**ためだ。

**受入条件**:
- **Given** AI 実行、**When** コード実行が必要、**Then** Cloud Run Jobs の一時インスタンス(リソース制限・ネットワーク制限付き)で実行される
- **Given** 実行完了、**When** ジョブが終了、**Then** サンドボックスは破棄され中間生成物のみ Cloud Storage に保存される
- **Given** サンドボックス、**When** 外部ネットワークアクセス、**Then** 許可リスト(GitHub, Anthropic API など)のみに制限される

---

### US-6.4: 依存パッケージの脆弱性は CI でスキャンされる [System, Security]
**優先度**: Must | **ペルソナ**: (システム内部)

**ストーリー**: **システム**として、**依存パッケージの脆弱性を CI で継続的にスキャン**したい。それは**既知の CVE を素早く検出・修正する**ためだ。

**受入条件**:
- **Given** CI の PR チェック、**When** Go は govulncheck、Python は pip-audit、Node は npm audit を実行、**Then** 高深刻度の脆弱性があれば CI 失敗する (NFR-S-05)
- **Given** 週次スキャン、**When** main ブランチで実行、**Then** 新たに発見された脆弱性をチケット化する

---

### US-6.5: ビジネスロジックにプロパティベーステストを適用する [System, PBT]
**優先度**: Must | **ペルソナ**: (システム内部)

**ストーリー**: **システム**として、**ビジネスロジック・データ変換・シリアライゼーションにプロパティベーステスト(PBT)を適用**したい。それは**エッジケースを自動的に発見する**ためだ。

**受入条件**:
- **Given** Go の BE コード、**When** gopter で PBT を書く、**Then** CI で実行される
- **Given** Python の AI サービス、**When** hypothesis で PBT を書く、**Then** CI で実行される
- **Given** TypeScript の FE 共有ロジック、**When** fast-check で PBT を書く、**Then** CI で実行される
- **Given** 80% カバレッジ、**When** CI 実行、**Then** カバレッジ未達は警告(NFR-T-02)

---

### US-6.6: AI リスク判定ロジックには決定的テストを適用する [System, PBT]
**優先度**: Must | **ペルソナ**: (システム内部)

**ストーリー**: **システム**として、**リスク判定のしきい値(差分行数、ファイル種別)について決定的なテストケースを用意**したい。それは**判定ロジック変更時の回帰を検知する**ためだ。

**受入条件**:
- **Given** リスク判定関数、**When** 「50 行以下 + 通常ファイル」の入力、**Then** "low" を返す
- **Given** 「auth.go を含む変更」の入力、**When** 判定、**Then** "high" を返す
- **Given** 判定ロジックを変更、**When** PR を作成、**Then** 既存テストケースに影響が出る場合は PR で可視化される
